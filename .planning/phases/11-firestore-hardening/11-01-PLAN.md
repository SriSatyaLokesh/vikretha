function isAuthorized(shopId) {
  let config = get(/databases/$(database)/documents/shops/$(shopId)/config/main);
  return request.auth != null
    && request.auth.token.email != null
    && request.auth.token.email in config.data.authorized_emails;
}
match /shops/{shopId}/{document=**} {
  allow read: if isAuthorized(shopId);
  allow write: if isAuthorized(shopId);
}

  <action>
Replace the entire content of `firestore.rules` with the v2 rules from ADR-0001. Key changes:

1. **Helper functions:**
   - `getConfig(shopId)` — returns config doc data
   - `isAuthenticated()` — checks auth != null and email != null
   - `isAuthorized(shopId)` — checks email in authorized_emails
   - `getRole(shopId)` — returns staff_roles[email]
   - `isOwnerOrAdmin(shopId)` — authorized + role in ['owner', 'admin']

2. **Per-collection rules:**
   - `/shops/{shopId}/config/main`:
     - read: isAuthorized
     - write: isAuthorized AND getRole == 'owner'
     - create (bootstrap): auth != null AND email in request.resource.data.authorized_emails AND request.resource.data.staff_roles[email] == 'owner'
   - `/shops/{shopId}/inventory/{itemId}`:
     - read: isAuthorized
     - create/write (full): isOwnerOrAdmin AND data has ['name','unit','price','stock'] AND price > 0 AND stock >= 0
     - update (stock-only): isAuthorized AND diff.affectedKeys().hasOnly(['stock']) AND stock is number
   - `/shops/{shopId}/sales/{saleId}`:
     - read: isAuthorized
     - create: isAuthorized AND data has ['saleId','timestamp','items','total'] AND total >= 0
     - update, delete: false (immutable)
   - `/shops/{shopId}/counters/{counterId}`: read/write if isAuthorized
   - `/shops/{shopId}/daily_summary/{dateId}`: read/write if isAuthorized
   - `/shops/{shopId}/monthly_summary/{monthId}`: read/write if isAuthorized
   - `/shops/{shopId}/summary/{docId}`: read/write if isAuthorized (legacy, keep working during migration)

3. Keep the instructional comments at the top (SETUP INSTRUCTIONS, HOW IT WORKS, ADDING STAFF) updated to reflect role-based model.

**IMPORTANT:** The inventory write rule must have TWO match paths:
- A general `allow write` for owner/admin with full validation
- A separate `allow update` for any authorized user (cashier) that ONLY allows `stock` field changes

This is because Firestore evaluates rules with OR logic — if either matches, access is granted.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → Firestore | All writes validated by security rules server-side |
| staff → owner data | Role enforcement prevents privilege escalation |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-11-01 | Elevation of Privilege | Staff removes owner from authorized_emails | mitigate | Config write restricted to owner role only |
| T-11-02 | Tampering | Cashier modifies inventory price/name | mitigate | Inventory write requires owner/admin; cashier can only update `stock` field |
| T-11-03 | Tampering | User deletes/modifies sale records | mitigate | Sales are immutable: `allow update, delete: if false` |
| T-11-04 | Tampering | Invalid data written (negative prices, missing fields) | mitigate | Write validation in rules: `price > 0`, `stock >= 0`, required field checks |
| T-11-05 | Information Disclosure | Summary docs readable by all staff | accept | All authorized users need dashboard access; no PII in summaries |
</threat_model>
